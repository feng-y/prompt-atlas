# Prompt Studio 训练指南

> 目标：通过真实 Prompt Pair 训练表达能力，把离散需求描述逐步提升为高密度、自然、可验证的 Decision Context。

## 1. 当前训练对象

Prompt Studio 不以复杂 Prompt Engineering、固定模板或新增 Skill 为主要目标。

当前需要纠正的旧行为是：

```text
需求 A
需求 B
还要 C
注意 D
输出 E
```

这类 Prompt 通常信息不少，但模型仍需自行推断：为什么现在做、哪个问题最关键、哪些边界不能突破、本轮停在哪里、什么证据才算完成。

训练后的目标表达是：

```text
为什么现在做
当前真实状态与核心困惑
允许与禁止的边界
本轮唯一交付
可观察的完成证据
```

这不是要求每条 Prompt 都机械包含五段，而是训练这些信息在需要时能够自然出现。

## 2. 当前优势与短板

### 优势

- 能准确识别真实工程问题，需求信息丰富。
- 对架构边界、验证风险和方案漂移较敏感。
- 能快速指出模型输出中的核心偏差，不满足于表面润色。
- 偏好最小改写、真实场景和弱启发式控制。

### 当前短板

- Prompt 偏离散，常以连续补充需求的方式形成。
- “做什么”通常明确，“为什么这是当前真正的问题”出现较晚。
- Research、Design、Implementation、Verification 容易在一轮中混合。
- Boundary 有时只有禁止项，没有给出推荐替代路径。
- Success 容易停留在“完整、统一、对齐”等自证式描述，而不是工具结果或 repo evidence。

## 3. 五类学习基线

这五类不是人物崇拜，也不是要求复制措辞。它们分别补足一种表达能力。

### 3.1 Anthropic / Alex Albert：Decision Context

**先训练这一类。**

学习重点：

- Why：任务服务于什么更大的目标。
- Boundary：本轮允许做什么，不允许做什么。
- Immediate Task：现在只完成哪一个交付。
- Evidence：何种证据支持进展或完成声明。
- Subtraction：减少为旧模型补偿而写死的过程。

#### 旧表达

```text
研究 Wayfinder，看看 setup，结合 DaVinci，设计方案。
```

#### 最小改写

```text
DaVinci 的执行与验证能力已经相对明确，当前薄弱点是 Wayfinder 发现 unknown 后仍依赖模型自行判断去哪里求证。

研究 Wayfinder 的现有 setup 能否建立 repo-local binding，让它使用 DaVinci 已有的架构、knowledge routing 和 replay surface。

本轮只形成设计判断，不 fork Wayfinder，也不进入实现。
```

#### 可迁移原则

先让模型知道它正在帮助解决什么决策问题，再给任务。边界不是附注，而是任务定义的一部分。

---

### 3.2 Matt Pocock：少量强概念控制多轮行为

学习重点：

- 用 Destination 表达长期目标。
- 用 Fog 区分还无法准确提问的未知区域。
- 用 Frontier 限定当前可推进的决策。
- 用 Grill 让事实由 Agent 查明、取舍由人决定。
- 用 Review against Spec 区分“实现合理”与“符合原始意图”。

#### 示例

```text
这项工作当前还不能稳定写成 SPEC。

先明确 destination，再识别已经能够准确表述的决策问题和仍处于 fog 中的部分。不要把看不清的问题强行拆成实现任务。

本轮只建立共享地图和当前 frontier，不开始交付最终改动。
```

#### 可迁移原则

Prompt 不一定需要更多步骤。几个边界清楚的领域概念，往往比长流程更能稳定控制多轮工作。

---

### 3.3 Simon Willison：事实、证据、推断与实验

学习重点：

- 先构造最小复现或反馈环，再解释原因。
- 区分代码直接证明的事实、由事实支持的推断、尚未验证的假设。
- 优先使用官方文档、源码、规范、发布记录或可复现实验。
- 来源冲突时保留冲突，不为了答案整齐而自行合并。

#### 示例

```text
不要先解释这个行为为什么发生。

先找到能够稳定复现它的最小输入或测试，确认当前版本确实失败；然后一次只改变一个变量并记录结果。

最后分别列出：实验已经确认的事实、最可能的解释、仍未排除的替代原因。
```

#### 可迁移原则

Prompt 要约束“结论如何获得”，而不仅是“输出什么结论”。

---

### 3.4 Lilian Weng：从单次 Prompt 到 Harness 改进

学习重点：

- 不只修复本次失败，判断失败发生在模型、上下文、路由、工具、验证反馈还是停止条件。
- Harness 改造需要假设、真实 probe、基线、指标和 retain/rollback。
- 单个 case 默认只是 hypothesis，不自动晋升为长期规则。
- 稳定知识需要明确未来消费者和权威落点。

#### 示例

```text
不要只修复这一次 Agent 的错误。

判断失败主要发生在哪一层：上下文缺失、知识路由错误、工具能力不足、验证反馈不足，还是停止条件不清。

提出最小 harness 改动，并用两个真实 case 对比改造前后的路由准确率、修复轮次、人工干预和证据完整性。没有改善则回滚。
```

#### 可迁移原则

当同类 Prompt 反复需要补充相同信息时，问题可能不再属于单次 Prompt，而属于长期 Harness。

---

### 3.5 Boris Cherny：任务清晰后进入 Goal / Loop

**这一类最后训练。**

学习重点：

- Loop 必须有 Goal、可观测反馈与停止条件。
- 每轮重新读取最新状态，不沿用过时假设。
- 并行 Agent 适合独立调查，不适合同时设计整套系统。
- 只有任务边界和 verification 已经清晰时，才值得 Loop 化。

#### 示例

```text
目标是让当前 slice 满足 SPEC，并通过指定 replay。

每轮读取最新代码和验证结果，选择影响最大的一个偏差，做最小修改并重跑验证。

只要失败仍能由现有证据解决就继续；验证全部通过，或出现会改变目标、范围或核心取舍的人类决策时停止。
```

#### 可迁移原则

“持续推进”不是 Loop。没有状态、反馈、PASS/RETRY/BLOCK 与 stop condition 的循环只是重复。

## 4. 推荐训练顺序

### 阶段一：Prompt Pair 仿写

目标：刷新离散需求型表达。

每次使用一个真实工作 Prompt：

1. 保留原始 Prompt，不改写 source。
2. 只指出影响最大的 1–3 个问题。
3. 给出 Minimal Rewrite，优先补最关键的一处。
4. 再给一个 Expert Prompt 展示表达上限。
5. 用户仿写同类 Prompt，比较差异。

进入下一阶段的信号：用户能够在不套模板的情况下，自然补出 Why、Boundary 或 Immediate Task。

### 阶段二：聚焦单项能力

分别训练：

- Why / Decision Context
- Boundary 与替代路径
- Immediate Task
- Fact / Inference / Hypothesis
- Observable Success

每轮只训练一个变量，不同时纠正所有问题。

### 阶段三：盲写与反向拆解

- 给出真实需求，由用户独立写 Prompt。
- Review 时先让用户判断最薄弱的一处。
- 拆解 Anthropic、Matt 或工程案例：每句话解决什么问题，删除后会发生什么。

### 阶段四：真实 Outcome 验证

记录 Prompt 实际交给模型后的结果：

- 是否更快找到 repo anchor；
- 是否减少 repair rounds；
- 是否降低人工补充次数；
- 是否出现过度执行；
- 是否产生更完整的验证证据。

只有真实 outcome 支持的规律，才提升到 principles 或 patterns。

## 5. 默认 Review 合同

当用户提交一个 Prompt 时，默认先 review Prompt 本身，而不是直接完成任务。

输出顺序：

1. **Prompt 判断**：当前是否已经足够好。
2. **最关键问题**：最多三个；必须指出“只改一个地方时最值得改什么”。
3. **Minimal Rewrite**：保留用户目标与风格。
4. **为什么变好**：说明减少了哪一部分模型猜测或决策成本。
5. **本次刻意练习**：只给一个训练目标。

不要为了形式统一而润色；如果 Prompt 已经足够清楚，直接说明无需修改。

## 6. 会话复盘 Prompt

```text
请 review 我在这个会话中编写 Prompt 的方式，而不是 review AI 的回答或任务本身。

先识别我本轮 Prompt 的主要表达模式，再结合实际对话结果判断哪些表达有效，哪些导致模型猜测、发散、过度执行或需要我反复纠正。

重点检查：
- 我是否建立了足够的 Why / Decision Context；
- 背景是否帮助决策，还是只是离散信息堆积；
- Boundary 是否清楚，并给出合理替代路径；
- Immediate Task 是否聚焦；
- Success 是否可以由证据观察；
- 哪些追加说明本可以更早进入首个 Prompt；
- 哪些约束其实没有必要，限制了模型判断。

请输出：
1. 本会话做得最好的两处；
2. 出现频率最高的三个问题；
3. 一组具有代表性的“原始 Prompt → 最小改写”；
4. 如果只能改变一个习惯，下一次最值得练习什么；
5. 一道基于本会话真实场景的仿写题。

不要一次提供过多理论，也不要为了修改而修改。明确区分：从 Prompt 文本直接观察到的事实、根据模型表现形成的推断、仍需更多会话验证的假设。
```
