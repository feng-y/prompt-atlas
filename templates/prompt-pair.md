## Current Prompt

> 粘贴实际准备发送或已经发送的 Prompt。保留原貌，不先润色。

## Improvement

用简短文字记录这次提升过程：原 Prompt 最影响结果的问题是什么，改版后最关键的变化是什么。

重点记录真实变化，不逐项展开完整 review。Foundation、Task Contract、自治边界和工具路由等分析方法由 Guide 承担；case 只保留理解这次改版所必需的内容。

## Minimal Rewrite

> 尽量保持原表达，只完成最小但有价值的改版。它可以用于观察单一变化，也可以作为从原 Prompt 到 Final Prompt 的中间版本。

## Why It Is Better

用一两句话说明这次改版为什么更可能改善结果。优先说明减少了什么关键猜测、冲突或无效控制，不使用“更清晰、更专业”一类空泛评价。

## Transferable Principle

一句可迁移原则。未经多个真实任务验证时，标记为 candidate，不直接提升为仓库 principle。

## Outcome（执行后补）

- 实际发送版本：
- 模型结果：
- 返修轮次：
- 证据 / verdict：
- 下次只保留什么：

## Final Prompt

Final Prompt 必须是页面最后一节，也是唯一可直接发送给目标模型的版本。

写作前可使用 [Prompt Foundation Structure](../guides/prompt-foundation-structure.md) 做内部检查，但不要把 review 维度机械写成最终 Prompt 的段落标题，也不要在 case 中逐项复述完整分析。

在这里写一份自然、连续、可直接复制的 Markdown。描述目的地、必要上下文、真实边界与完成标准；删除不会改变行为的过程脚手架。只有任务本身需要正式合同、结构化返回或 Harness 消费时，才采用显式栏目或列表。

不得混入分析、比较说明、多个备选版本或“见上文”式引用。
